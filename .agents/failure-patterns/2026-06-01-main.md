# 2026-06-01 — Nix Rust package/update regressions

## When this applies

Use this checklist when changing `package.nix`, `flake.nix`, `flake.lock`, `Cargo.lock`, or `scripts/update-package.sh` for the Codex Rust package.

## Failure pattern

A package update can appear correctly pinned while still regressing CI or supply-chain safety if it:

- lets Nix fetch crates through the flaky crates.io API download path instead of the static crate mirror;
- replaces `rustPlatform.buildRustPackage`'s default build phase just to clean generated Cargo config, thereby skipping default hooks/wrappers;
- computes `src.hash` from one upstream archive but copies `Cargo.lock` from a second download; or
- updates Codex without syncing `rustyV8Version` and both Linux `rusty_v8` archive hashes to the locked `v8` crate version.

## Required prevention

- Keep `cargoLock.extraRegistries` mapping `https://github.com/rust-lang/crates.io-index` to `https://static.crates.io/crates` unless a replacement is validated in CI.
- If generated Cargo config needs cleanup, prefer hook phases such as `preBuild`; do not add a custom `buildPhase` without explicitly proving the default `buildRustPackage` hooks remain covered.
- In `scripts/update-package.sh`, copy `Cargo.lock` from the Nix store path returned by the same `nix store prefetch-file --json --unpack` call used to compute the source hash. Avoid a second `curl`/`tar` source download.
- Parse the locked `v8` crate version and prefetch both supported Linux `rusty_v8` archives before updating `package.nix`.

## Verification before hand-off

Run the repository gate plus updater idempotence:

- `just lint`
- `nix flake check --all-systems 'path:.'`
- `nix build -L 'path:.#codex'`
- `nix run 'path:.#codex' -- --version`
- `./scripts/update-package.sh --version <current-package-version>`
